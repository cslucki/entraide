## 1. Purpose

Define the tenant resolution strategy for the root domain (`test.laravel` / `bouclepro.com`).

This document answers all 12 resolution questions from T075.0 findings, defines a route matrix, and produces a sequenced task breakdown for implementation.

**Rule:** Strategy-first. No `app/`, `routes/`, `resources/`, `database/`, `config/` modifications in this phase.

---

## 2. Problem Statement

The T075.0 audit (docs/audits/T075.0-organization-native-tenant-audit.md, commit f7c0db3) identified all business routes on the root domain operate **without any tenant resolution middleware**.

| Route Group        | Middleware              | Tenant Bound | Risk |
|--------------------|-------------------------|-------------|------|
| Auth routes        | guest/auth              | N/A (auth)  | None |
| `/` (landing)      | web                     | No          | P1   |
| `/dashboard`       | web, auth               | No          | P0   |
| `/membres`         | web                     | No          | P0   |
| `/echanges`        | web                     | No          | P0   |
| `/boucles`         | web                     | No          | P1   |
| `/services/*`      | web                     | No          | P0   |
| `/requests/*`      | web                     | No          | P0   |
| `/transactions/*`  | web                     | No          | P0   |
| `/messages/*`      | web, auth               | No          | P0   |
| `/blog/*`          | web                     | No          | P2   |
| `/loops/*`         | web                     | No          | P0   |
| `/explorer`        | web                     | No          | P1   |
| `/search`          | web                     | No          | P1   |
| `/admin/*`         | web, auth, admin        | No (global) | None |
| `/org/{org}/*`     | ResolveCommunity        | Yes         | None |

The core issue: `ResolveCommunity` middleware only activates under `/{community}` prefix. `CurrentOrganization::get()` checks `app('current_organization')` then `app('current_community')` — both are **null** on the root domain. `BelongsToTenantScope` silently skips when no tenant is resolved, producing no WHERE clause.

Architecture rule: **Organization = Tenant. Root domain is NOT tenantless.**

## 3. The 12 Questions — Answers

### Q1. What should `/` (landing) do?

**Answer:** Public Platform landing page. Unauthenticated users see platform stats, hero. Authenticated users with an Organization see the same landing (no redirect).

- `/` is intentionally tenantless as a Platform page.
- No Organization resolution needed.
- Public data only (total users, total active services, total completed exchanges).
- Stats must be aggregated/anonymized — no per-Organization data.
- Featured services cross-Organization: NOT validated until an explicit public publication strategy exists.
- **P1 remains open.** Not downgraded.

### Q2. What should `/dashboard` do?

**Answer:** Tenant-scoped authenticated user dashboard. MUST require an Organization context.

- If authenticated + Organization resolved → show dashboard (current behavior but tenant-filtered).
- If authenticated + NO Organization → redirect to `/onboarding/organization`.
- If unauthenticated → redirect to `/login`.

The dashboard must:
- Show ONLY the user's data within their Organization.
- Use `$user->community_id` as the default Organization resolution source.
- Never expose cross-Organization transactions, services, messages.

### Q3. What should `/membres`, `/echanges`, `/boucles` do?

**Answer:** All require Organization resolution.

- **`/membres` (P0):** Currently dumps ALL users in the platform. Must be scoped to Organization members only.
- **`/echanges` (P0):** Currently shows cross-org transactions. Must be scoped to Organization exchanges.
- **`/boucles` (P1):** Legacy route. Historically listed communities/partners co-branded. NOT the future Loop concept. NOT a tenant. NOT a canonical future route.

Strategy:
- `/membres` → Organization-scoped: requires resolved Organization, redirect if missing.
- `/echanges` → Organization-scoped: requires resolved Organization, redirect if missing.
- `/boucles` → Legacy route to deprecate. Target landing is `/partners`. `/boucles` may later redirect to `/partners`. The old `/boucles/creer` form must be audited and likely repositioned as "Devenir partenaire". No public Loop creation on root domain.

### Q4. What should `/services/*`, `/requests/*`, `/transactions/*`, `/messages/*`, `/blog/*`, `/loops/*` do?

**Answer:**

| Route           | Strategy                           |
|-----------------|------------------------------------|
| `/services/*`   | Organization-scoped. Must filter by resolved Org. |
| `/requests/*`   | Organization-scoped. Must filter by resolved Org. |
| `/transactions/*`| Organization-scoped. Must filter by resolved Org. |
| `/messages/*`   | Organization-scoped. Only show messages within Org. |
| `/blog/*`       | Organization-scopé. `/blog` = Organization par défaut de la plateforme. `/{partnerSlug}/blog` = Organization partenaire. |
| `/loops/*`      | Organization-scoped. Loops belong to an Org. |

Blog : Organization-scopé (AD06). Plus de débat global OU Organization-scopé. T075.5 doit corriger BlogPost tenantless, scoper index/show/create/update, et ajouter les tests.

### Q5. How should the system resolve the Organization for an authenticated user?

**Answer:** Cela dépend du **contexte URL**. Il n'y a pas une unique résolution pour tous les users authentifiés.

1. **Routes personnelles authentifiées** (`/dashboard`) → résolvent l'Organization du user connecté.
   - Session `resolved_organization_id`.
   - Fallback `auth()->user()->community_id` (legacy).
   - Si aucune Organization : redirection vers onboarding.

2. **Routes root métier** (`/blog`, `/explorer`, `/membres`, `/loops`, etc.) → résolvent l'Organization par défaut de la plateforme.
   - Ne pas utiliser l'Organization du user connecté par défaut.
   - Ces routes sont publiques et affichent le contenu de l'Organization par défaut.

3. **Routes partner slug** (`/{partnerSlug}/{feature}`) → résolvent l'Organization liée au partnerSlug.
   - Mapping Partner → Organization.
   - Permet de voir les données d'une Organization partenaire sans y être connecté.

4. **Routes Platform globales** (`/`, `/login`, etc.) → aucune Organization résolue.

### Q6. What should happen for authenticated users without an Organization?

**Answer:** Show onboarding/org-creation. Block access to business routes.

- Redirect from: `/dashboard`, `/membres`, `/echanges`, `/services/*`, `/requests/*`, `/transactions/*`, `/messages/*`, `/loops/*`.
- Allowed: `/`, `/logout`, profile settings. `/blog` Organization-scopé (T075.5).
- Create `/onboarding/organization` page (future task) — simple form to create first Organization.

### Q7. What should unauthenticated visitors see?

**Answer:** Only global/public content.

- Allowed: `/`, `/login`, `/register`, `/password/*`, `/mentions-legales`, `/sitemap.xml`. `/blog` Organization-scopé (T075.5).
- Blocked (redirect to login): `/dashboard`, `/membres`, `/echanges`, `/services/*`, `/requests/*`, `/transactions/*`, `/messages/*`, `/loops/*`, `/explorer`, `/search`.
- `/membres` currently lacks `auth` — needs review. `/boucles` is legacy and will be deprecated.

Confirmed global routes:

Pending decisions:

| `/search`          | Needs Organization context       |

| `/boucles`         | Legacy. Target: `/partners`.     |

### Q9. Why is `/admin/*` intentionally global?

T075.0 established: Admin is **platform-level**, not tenant-level. Admins manage all Organizations, all users, all transactions. Admin routes intentionally bypass `BelongsToTenantScope`.

This is correct architecture and remains unchanged.

### Q10. Should the root domain redirect to a default Organization?

**Answer:** No URL redirect. **Implicit resolution only.**

The root domain stays at `/` — the system resolves the Organization **invisibly** via middleware. The user stays on the same URL path. The middleware binds the Organization to the container and session.

Exception: if user has no Organization and lands on a tenant-required route, redirect to `/onboarding/organization`.

### Q11. Should the root domain resolve a default Organization?

**Answer:** Yes. Implicit resolution via middleware (T075.2), par contexte URL — ne pas réduire à une simple session + user.community_id.

Ordre de résolution par contexte URL :

1. **Platform global route** — aucune Organization requise. Routes : `/`, `/login`, `/register`, `/password/*`, `/mentions-legales`, `/sitemap.xml`, `/partners`, `/admin/*`.

2. **Default Organization route (`/{feature}`)** — résout l'Organization par défaut de la plateforme. Routes : `/blog`, `/explorer`, `/membres`, `/loops`, `/services`, `/requests`, `/messages`, etc.

3. **Partner slug route (`/{partnerSlug}/{feature}`)** — résout l'Organization liée au partnerSlug via mapping Partner → Organization. Routes : `/{partnerSlug}/blog`, `/{partnerSlug}/explorer`, etc.

4. **Authenticated personal route** — résout l'Organization du user connecté :
   - Session key `resolved_organization_id`.
   - Fallback `auth()->user()->community_id` (legacy).
   - Redirection vers onboarding si aucune Organization.

5. **Fail-safe** — blocage / redirection / 404 si aucune Organization résolue sur route métier.

Ce choix évite la complexité sous-domaine, préserve les bookmarks existants, et distingue clairement les contextes de résolution.

### Q12. Should we activate `/org/{organization}` now or later?

**Answer:** Later (T075.3+). NOT in T075.2.

`/org/{organization}` routes are a separate concern — explicit Organization routing. Activating them now would require:
- Duplicating all route definitions.
- View/component URL updates.
- Test rewrites.
- Livewire hydration regressions.

Focus T075.2 on **implicit resolution only**. `/org/{org}` explicit routing comes in a subsequent phase.

## 5. Partner / Co-branding Terminology

### Definitions

| Term        | Definition                                              |
|-------------|---------------------------------------------------------|
| Platform    | Système global BouclePro. Toutes les routes globales.   |
| Partner     | Entrée de co-branding / distribution / partenariat.     |
| Organization| Tenant. Frontière de sécurité, billing, gouvernance.    |
| Loop        | Groupe collaboratif interne à une Organization.         |
| Member      | Utilisateur appartenant à une Organization.             |

### Rules

- **Partner n'est pas un tenant.** Un Partner est une entrée publique, pas une frontière de sécurité.
- **Loop n'est pas un tenant.** Les Loops appartiennent à une Organization. C'est l'Organization qui est le tenant.
- **Organization = Tenant.** La seule frontière de sécurité et d'isolation.
- Aucune donnée métier Organization-scopée ne doit être exposée sans Organization résolue.

### Routes partenaires conceptuelles

| Route                      | Rôle                                                |
|----------------------------|-----------------------------------------------------|
| `/partners`                | Liste publique des partenaires co-brandés. Cible de remplacement de `/boucles`. |
| `/partners/{partnerSlug}`  | Route structurée pour une page partenaire.          |
| `/{partnerSlug}` (ex: /bni)| Alias court / vanity URL publique. Peut être l'URL canonique. |
| `/partners/bni` → `/bni`   | Redirection recommandée (évite duplicate SEO).      |
| `/bni/loops`               | Loops internes de l'Organization BNI. Organization-scopé. |

### Relation Partner → Organization

- Un Partner peut pointer vers une Organization (ex: BNI = Organization + Partner).
- Mais un Partner n'est pas obligatoirement une Organization (ex: partenaire purement marketing sans tenant).
- Les Loops appartiennent toujours à une Organization, jamais à un Partner directement.

### Legacy `/boucles`

- `/boucles` est une route legacy historique.
- L'écran "Les Boucles" mélangeait communities, partenaires, et création d'espace.
- L'ancien bouton "Créer ma boucle" n'est plus adapté.
- L'ancien formulaire `/boucles/creer` doit être audité : pourrait devenir "Devenir partenaire".
- **Ne pas confondre `/boucles` legacy avec le futur concept Loop.**
- La cible UI dans le header est "Partenaires", pas "Boucles".
- Aucune création publique de Loop ne doit exister sur le root domain.

### Règle absolue

**Toutes les fonctionnalités métier actuelles et futures de BouclePro doivent être Organization-scopées.**

Aucune feature métier ne doit fonctionner sans Organization résolue.
Une route Platform globale doit être une exception explicite, documentée, non métier, ou purement marketing/légale/auth/admin.

| `/admin/*`           | Platform admin (global intentionnel) |

**Règle :** Les futures features ne doivent jamais être ajoutées comme routes Platform globales par défaut. Toute exception doit être explicitement documentée et validée.

### URL Context Resolution Order

Ordre de résolution de l'Organization selon le contexte URL :

#### 1. Platform global route
- Routes : `/`, `/login`, `/register`, `/password/*`, `/mentions-legales`, `/sitemap.xml`, `/partners`, `/admin/*`.
- Comportement : **aucune Organization requise**.
- `/admin/*` reste accessible sans Organization (Platform admin global intentionnel).
- Aucune feature métier Organization-scopée sur ces routes.

#### 2. Default Organization route
- Routes : `/blog`, `/explorer`, `/membres`, `/loops`, `/services`, `/requests`, `/messages`, et toute autre feature métier sans partner slug.
- Comportement : **résout l'Organization par défaut de la plateforme**.
- Le mécanisme de résolution peut être :
  - session utilisateur (si déjà choisie),
  - `user.community_id` (legacy, temporaire),
  - configuration Platform (si Organization par défaut définie pour la plateforme).
- À définir dans T075.2.

#### 3. Partner slug route
- Routes : `/{partnerSlug}`, `/{partnerSlug}/blog`, `/{partnerSlug}/explorer`, `/{partnerSlug}/membres`, `/{partnerSlug}/loops`, etc.
- Comportement : **résout l'Organization liée au partnerSlug**.
- Le partner slug est résolu via une table de mapping Partner → Organization.
- Permet de visualiser les données d'une Organization partenaire sans y être connecté.

#### 4. Authenticated personal route
- Routes : `/dashboard` notamment.
- Comportement : **résout l'Organization du user connecté**.
- Si l'utilisateur n'a pas d'Organization : **redirection vers onboarding**.
- Contrairement aux routes "Default Organization", ces routes sont strictement personnelles.

#### 5. Fail-safe
- Toute route métier sans Organization résolue doit **bloquer, rediriger ou retourner 404** selon le contexte.
- Aucune donnée métier ne doit être chargée sans Organization résolue.
- `BelongsToTenantScope` ne doit jamais être silencieusement sauté (T075.7).

---

### Rappel conceptuel

- **Partner** n'est pas un tenant. C'est une entrée co-branding / distribution.
- **Organization** reste le tenant. Frontière de sécurité unique.
- **Loop** reste un groupe interne à une Organization.
- **Community** / `community_id` / `current_community` reste legacy technique temporaire.

## 8. Task Breakdown

### T075.2 — URL Context Resolver Middleware minimal
*Dependencies: none*
*Files: `app/Http/Middleware/ResolveTenant.php`, `routes/web.php`, `app/Http/Kernel.php`*

Créer un middleware `ResolveTenant` (à confirmer) qui classifie le contexte URL et résout l'Organization selon l'ordre défini en section 6 :

1. **Platform global route** — aucune résolution d'Organization, middleware non activé.
2. **Default Organization route** (`/blog`, `/explorer`, etc.) — résout l'Organization par défaut de la plateforme.
3. **Partner slug route** (`/{partnerSlug}/...`) — résout l'Organization liée au partnerSlug via mapping (implémentation minimale, pas de modèle Partner).
4. **Authenticated personal route** (`/dashboard`) — résout l'Organization du user connecté :
   - Session → `app('current_organization')`.
   - Fallback sur `auth()->user()->community_id`.
   - Bind dans `app('current_organization')`.
   - Stocke en session.
   - Redirige vers `/onboarding/organization` si Org requise mais absente.
5. **Fail-safe** — blocage / redirection / 404 si une route métier requiert Organization et qu'aucune n'est résolue.

**Périmètre strict (ne rien faire de plus) :**
- Pas de création du modèle Partner.
- Pas d'implémentation complète de l'architecture Partner.
- Pas de fix BlogPost.
- Pas de hidden `community_id` fix.
- Pas de Policies.
- Pas d'API.
- Pas de BelongsToTenantScope hardening (sauf log non bloquant si validé plus tard).
- Pas de renommage `/boucles`.
- Pas de création `/partners`.
- Pas de migration DB.
- Pas de sous-domaine.

**Acceptance criteria:**
- Middleware classifie le contexte URL (Platform global / Default Org / Partner slug / Authenticated personal).
- Bind `current_organization` pour les routes métier existantes.
- Fail-safe si route métier requiert Organization et aucune résolue.
- Platform global routes exemptées de résolution.
- Root métier routes résolvent Organization par défaut.
- Partner slug routes résolvent Organization partenaire.
- Authenticated personal routes résolvent Organization du user connecté.
- User sans Organization redirigé vers onboarding.
- Unauthenticated user cannot access tenant-required routes (auth already present or added).
- No changes to existing controller logic.
- All existing tests pass.

## 12. Architecture Decisions

| ID   | Decision                                         | Rationale                                              |
|------|--------------------------------------------------|--------------------------------------------------------|
| AD01 | Implicit resolution, not URL-based               | No URL changes, no bookmark breakage, no subdomain     |
| AD02 | Session-cached Org resolution                    | Avoids DB query on every request                       |
| AD03 | User `community_id` as fallback source           | No new column needed, existing data works               |
| AD04 | `/membres` and `/loops` gain `auth`              | These expose member/loop data — must be authenticated   |
| AD05 | `/admin` stays global                            | Platform admin, not tenant admin                        |
| AD06 | Blog Organization-scopé                           | `/blog` = Organization par défaut. `/{partnerSlug}/blog` = Organization partenaire |
| AD07 | `/boucles` = legacy à déprécier                  | Cible `/partners`. Pas le concept Loop. Pas un tenant.  |
| AD08 | BelongsToTenantScope fails-closed (configurable) | No silent cross-org data leaks                          |
| AD09 | `/org/{org}` explicit routing deferred           | Doubles route surface area — do after implicit works    |
| AD10 | New middleware class (not modifying existing)    | Don't touch ResolveCommunity/ResolveOrganization now    |
| AD11 | Toute feature métier actuelle et future est Organization-scopée | Aucune feature métier sur route Platform globale par défaut |

## 15. Summary

The root domain Organization resolution strategy is **implicit, middleware-driven, and session-cached**.

No URL changes, no subdomains, no database migrations, no config changes in this phase.

**Concepts clarifiés après review COCKPIT :**
- `/boucles` = legacy à déprécier. Cible : `/partners`. Pas le concept Loop.
- Loop = groupe collaboratif interne à une Organization. N'est pas un tenant.
- Partner = entrée co-branding / distribution. N'est pas un tenant.
- Organization = Tenant. Frontière de sécurité unique.
- `/blog/*` = Organization-scopé. `/blog` = Organization par défaut. `/{partnerSlug}/blog` = Organization partenaire.
- `/` = page Platform publique. Stats agrégées uniquement. Featured services cross-org non validés.

Implementation order: T075.2 → T075.3 → T075.4 → T075.5 → T075.6 → T075.7 → T075.8 → T075.9 → T075.10 → T075.11.
