## Philosophie Produit

Cette doctrine documente les concepts fondamentaux du système Boucles Product pour BouclePro.

---

## Concepts Fondamentaux

### Flux Vivant

**Le Flux révèle le vivant.**

Le Flux représente l'ensemble des activités vivantes au sein d'une Organization. Il capture ce qui se passe réellement, en temps réel, au-delà des intentions et des projections.

**Caractéristiques:**
- Agnostique aux structures formelles
- Émergent par définition
- Révèle l'activité réelle vs déclarée
- Non hiérarchique
- Always-on

**Le Flux n'est pas:**
- Un workflow structuré
- Une séquence d'étapes
- Un arbre décisionnel
- Un système de catégorisation

**Le Flux, c'est:**
- Ce qui se passe
- Comment cela se passe
- Qui le fait
- Quand cela arrive

---

### Signaux

**Les Signaux révèlent les dynamiques émergentes.**

Les Signaux sont des patterns détectables dans le Flux. Ils représentent des tendances, des accélérations, des ralentissements, des réponses, des activations.

**Caractéristiques:**
- Déduits du Flux
- Temporels (avant, pendant, après)
- Relatifs (croissance, décroissance)
- Orientés action (réponses, participations)
- Emergents (non prédéfinis)

**Types de Signaux:**
- **Activation:** Une boucle s'anime (nouveau contenu, nouvelle interaction)
- **Réponse:** Un membre répond à un signal
- **Propagation:** Un signal se diffuse entre boucles
- **Intensification:** L'activité s'accélère dans une boucle
- **Épuisement:** L'activité ralentit naturellement

**Les Signaux ne sont pas:**
- Des notifications
- Des alertes
- Des règles métier
- Des actions déclenchées automatiquement

**Les Signaux, c'est:**
- Comprendre ce qui émerge
- Anticiper ce qui va se passer
- Identifier les patterns d'activité
- Voir l'invisible dans le Flux

---

### Boucles

**Les Boucles organisent l'énergie collective.**

Une Boucle est un espace collaboratif qui capture une intention collective. Elle fournit un contexte pour le Flux, un récepteur pour les Signaux, et une structure pour le Journal.

**Caractéristiques:**
- Contexte intentionnel (pourquoi cette boucle existe)
- Membres (qui participe)
- Visibilité (qui peut voir, qui peut rejoindre)
- État (active, pause, fermée)
- Organization-scoped ; tenant boundary = Organization. Une Boucle n'est jamais une frontière tenant.

**Une Boucle n'est pas:**
- Un groupe de discussion
- Un thread de messages
- Un projet ou tâche
- Une équipe ou département

**Une Boucle, c'est:**
- Un espace pour collaborer
- Un contexte pour l'énergie collective
- Un récepteur pour le Flux vivant
- Un conteneur pour le Journal narratif

**Relation avec autres concepts:**
- **Organization:** Tenant boundary, sécurité
- **Partner:** Co-branding/distribution, pas tenant
- **Loop ≠ Tenant:** Loop est interne à une Organization
- **Boucle = Loop:** Boucle est le terme produit pour Loop

---

### Décision `/boucles`

L'ancien usage `/boucles` comme demande de création de communauté est un legacy historique.

`/boucles` est désormais la route française canonique cible du vrai concept Boucles, conformément à `docs/specs/T077.0-boucles-product-surface-spec.md`.

Règles de lecture et d'affichage:

- Toute lecture ou affichage Boucles reste Organization-scopé.
- Aucune lecture globale des Boucles n'est autorisée.
- Toute surface publique Boucles doit résoudre une Organization ou fail-closed.
- Public ≠ Global.
- T077.0 et T077.4 remplacent l'ancienne lecture `/boucles` legacy pour les vraies Boucles.

### Journal de la Boucle

**Le Journal garde la mémoire narrative d'une Boucle.**

Le Journal est le récit séquentiel de ce qui s'est passé dans une Boucle. Il structure le Flux en narration intelligible pour les humains.

**Caractéristiques:**
- Séquentiel (timeline)
- Contextuel (lié à une Boucle)
- Narratif (compréhensible pour les humains)
- Consultable (recherche, navigation)
- Persistant (ne disparaît pas)

**Le Journal contient:**
- Activités (ce qui s'est passé)
- Signaux (patterns détectés)
- Contributions (contenu créé)
- Transitions (changements d'état)
- Moments clés (pics d'activité, décisions)

**Le Journal n'est pas:**
- Un historique de logs techniques
- Une archive brute du Flux
- Un système d'audit
- Un feed d'activité

**Le Journal, c'est:**
- La mémoire narrative humaine d'une Boucle
- Ce qui permet de comprendre l'histoire
- Ce qui permet de revenir en arrière
- Ce qui permet de transmettre le contexte

## Flux → Signaux → Boucles → Journal

### Relation Causale

```
[Flux Vivant]
    ↓ émergent
[Signaux] (patterns dans le Flux)
    ↓ capturés par
[Boucles] (contextes pour l'activité)
    ↓ mémorisés dans
[Journal] (narration séquentielle)
```

**Flux vivant:**
- Source de vérité de l'activité
- Non structuré, émergent
- Always-on

**Signaux:**
- Patterns détectés dans le Flux
- Indicateurs de dynamiques
- Base pour l'intelligence du système

**Boucles:**
- Contextes intentionnels
- Récepteurs du Flux
- Conteneurs pour le Journal

**Journal:**
- Mémoire narrative
- Séquence intelligible
- Outil de transmission

### Exemple Concret

**Flux vivant:**
- Marie crée un post
- Pierre répond
- Luc like la réponse
- Sophie ajoute un commentaire
- Marie répond à Sophie

**Signaux émergents:**
- Activation intense (boucle s'anime)
- Réponse rapide (< 5min)
- Propagation (3 membres impliqués)
- Intensification (5 interactions en 10min)

**Boucle:**
- Contexte: "Retour client Q4"
- Visibilité: Organization
- Membres: 12

**Journal de la Boucle:**
- 10:15 - Marie: Initial feedback sur X
- 10:17 - Pierre: Je confirme, j'ai vu le même
- 10:18 - Luc: 👍
- 10:20 - Sophie: Important à documenter
- 10:22 - Marie: Oui, je note ça

## Rôle de l'IA

### Principes

**L'IA aide à clarifier, relier et mémoriser sans remplacer l'élan humain.**

### Responsabilités IA

1. **Clarifier:**
   - Résumer des conversations complexes
   - Extraire des points clés
   - Identifier des décisions
   - Synthétiser des retours

2. **Relier:**
   - Connecter des conversations éparses
   - Identifier des patterns trans-boucles
   - Suggérer des liens pertinents
   - Détecter des thèmes récurrents

3. **Mémoriser:**
   - Structurer le Journal
   - Ajouter des métadonnées contextuelles
   - Faciliter la recherche
   - Préserver l'historique

4. **Ne PAS faire:**
   - Remplacer l'élan humain
   - Simuler des conversations
   - Créer de faux signaux
   - Décider à la place des membres

### Limites IA

**Ce que l'IA ne fait PAS:**
- Générer automatiquement du contenu
- Prendre des décisions pour les membres
- Simuler de l'activité humaine
- Remplacer la co-présence humaine
- Créer des boucles automatiquement

**Ce que l'IA fait:**
- Clarifier ce que les humains disent
- Relier ce que les humains font
- Mémoriser ce qui s'est passé
- Suggérer, pas décider

## Règles Anti-Dérive Conceptuelle

### 1. Flux ≠ Workflow

Le Flux est ce qui se passe, pas ce qui devrait se passer.

**CORRECT:**
- "Le Flux montre une accélération d'activité sur X"

**INCORRECT:**
- "Le Flux doit suivre le workflow Y"

### 2. Signaux ≠ Notifications

Les Signaux révèlent des patterns, pas alertent sur des événements.

**CORRECT:**
- "Signal d'activation intense détecté"

**INCORRECT:**
- "Notification: nouvelle activité détectée"

### 3. Boucles ≠ Groupes de Discussion

Une Boucle est un contexte intentionnel, pas un conteneur de messages.

**CORRECT:**
- "Boucle: Feedback Client Q4"

**INCORRECT:**
- "Groupe de discussion: retours clients"

### 4. Journal ≠ Logs

Le Journal est une mémoire narrative, pas un historique technique.

**CORRECT:**
- "Journal: Marie a proposé une amélioration X"

**INCORRECT:**
- "Log: POST /api/interactions 200 OK"

### 5. Organization = Tenant

Toute donnée Boucle doit être Organization-scopée.

**CORRECT:**
- "Boucle scoped à Organization X"

**INCORRECT:**
- "Boucle globale ou tenantless"

### 6. Loop ≠ Tenant

Loop est interne à Organization, pas un niveau de sécurité.

**CORRECT:**
- "Loop est un groupe collaboratif au sein d'Organization"

**INCORRECT:**
- "Loop est un tenant"

## Implications pour Tâches Futures

### T077.5 — Boucles Visibility Rules

Cette doctrine éclaire T077.5:
- La visibilité détermine qui peut voir le Flux d'une Boucle
- Les Signaux doivent respecter les règles de visibilité
- Le Journal doit être accessible selon la visibilité

**Décision T077.4:**
- Les Signaux ne doivent jamais exposer de données cross-organization
- Le Journal respecte strictement les scopes visibilité

### T077.6 — Boucles Membership & Roles

Cette doctrine éclaire T077.6:
- Les membres sont les participants actifs dans le Flux d'une Boucle
- Les roles déterminent ce que les membres peuvent faire avec le Flux

**Décision T077.4:**
- Les Signaux sont calculés sur les activités des membres
- Le Journal attribue les actions aux membres concernés

### T077.9 — Boucles Product Surface Runtime (V1)

Cette doctrine éclaire T077.9:
- La surface runtime expose le Flux, les Signaux, les Boucles, le Journal
- L'interface doit faire sentir le vivant du Flux
- Les Signaux doivent être visibles mais pas intrusifs

**Décision T077.4:**
- L'interface doit présenter le Flux comme vivant, pas statique
- Les Signaux doivent être des indicateurs subtils, pas des alertes
- Le Journal doit être facilement navigable mais pas encombrant

### T078.2 — Boucles Flux & Signaux Runtime

Cette doctrine fonde T078.2:
- T078.2 est la réalisation runtime de cette doctrine
- Le Flux vivant doit être capturé en temps réel
- Les Signaux doivent être calculés dynamiquement
- Le Journal doit être structuré automatiquement

**Décision T077.4:**
- T078.2 implémente cette doctrine sans la modifier
- T078.2 respecte les limites IA définies ici
- T078.2 ne peut pas dériver des concepts sans revenir à T077.4
