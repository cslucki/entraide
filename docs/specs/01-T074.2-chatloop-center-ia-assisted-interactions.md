## Executive Summary

T074.2 validates the product direction for ChatLoop as the living center of BouclePro.

ChatLoop is not a feature. It is the relational heart of the platform: the space where members ask for help, share knowledge, coordinate, and move forward together.

This specification consolidates:

- T074.0 — Technical audit of current messaging, mobile risks, Reverb readiness
- T074.1 — UX ergonomics for ChatLoop mobile, desktop and admin
- T074.1A — IA solution spike / FakeAIProvider / Lab IA admin
- T074-assets-index — Canonical UX reference images

**T074.2 does not implement anything.** It produces a codable specification for T074.3–T074.11. All decisions here are product decisions. No Laravel code, no routes, no Livewire components, no migrations, no Reverb, no OpenAI.

## Vision Produit

> "Je tape une intention floue, et BouclePro m'aide à créer une vraie demande d'aide envoyée au bon cercle."

### Phrase cœur

> "Qui peut m'aider ?" est une Interaction structurée / Demande d'aide, pas un simple message.

### Ce que BouclePro ChatLoop est

- Un espace d'entraide entre membres d'une Organization
- Un lieu de coordination calme et intentionnelle
- Un outil de progression: formuler un besoin, obtenir de l'aide, proposer son aide
- Un vecteur de transmission: savoirs, contacts, expériences
- Un mini-réseau de collaboration par Loop

### Ce que BouclePro ChatLoop n'est pas

- **Pas Slack** — pas de channels bruyants, pas de @here, pas de threads infinis
- **Pas WhatsApp** — pas de messagerie instantanée généraliste
- **Pas un chatbot** — l'IA clarifie, ne converse pas
- **Pas une marketplace type Malt** — pas de mise en relation freelance/client transactionnelle
- **Pas un réseau social** — pas de fil d'actualité, pas de likes, pas de viralité

## Concepts Stabilisés

| Concept | Statut | Règle |
|---|---|---|
| **Organization** | Canonique | Tenant boundary, security boundary, billing boundary |
| **Loop** | Canonique | Contexte collaboratif interne. N'est PAS un tenant. |
| **Member** | Canonique | Appartient à une Organization, peut rejoindre plusieurs Loops |
| **Interaction** | Canonique | Activité structurée: demande d'aide, proposition, message |
| **Community** | Legacy temporaire | `community_id` reste dans le code Laravel existant. Aucune migration T074.2. |
| **Platform** | Canonique | Infrastructure globale BouclePro |

### Règles absolues

- `Organization = Tenant` — confirmé. Jamais `Loop = Tenant`.
- `Loop ≠ Tenant` — confirmé. Loop est collaborative, pas une frontière de sécurité.
- `Community/community_id` — reste legacy technique. Aucune migration Community → Organization en T074.2.
- Aucune refonte de la base de données, des modèles ou des migrations existants.
- Aucune installation de package (composer ou npm).

## Parcours Cœur: "Qui peut m'aider ?"

### Parcours complet (intention → action)

```
1. Intention floue
   └── L'utilisateur tape une phrase: "J'ai besoin d'aide pour trouver mes premiers clients"

2. Clarification IA
   └── BouclePro reformule, structure, suggère une Loop

3. Demande structurée
   └── L'utilisateur voit un draft éditable: titre, description, type d'aide, Boucle conseillée

4. Boucle conseillée
   └── L'IA propose une Loop autorisée parmi les Loops dont le membre fait partie

5. Validation humaine
   └── L'utilisateur peut modifier, changer la Boucle, ou annuler

6. Publication
   └── L'utilisateur clique "Publier cette demande" → action métier normale

7. Feedback
   └── La demande apparaît dans la Loop. Les membres peuvent répondre.

8. Prochaine action utile
   └── L'IA suggère une prochaine étape (optionnel)
```

### Règles du parcours

- "Qui peut m'aider ?" **n'est pas un message libre.** C'est une Interaction structurée / Demande d'aide.
- Le texte utilisateur devient **un draft éditable**, pas un message publié.
- **Aucune publication automatique.** Rien n'est publié sans validation humaine explicite.
- L'utilisateur peut à tout moment modifier le titre, la description, la Boucle cible.
- Si l'utilisateur n'est pas satisfait du draft, il peut revenir à l'input initial.

## Rôle Exact de l'IA

### L'IA peut (et doit)

| Capacité | Description |
|---|---|
| Clarifier | Transforme une intention floue en besoin exprimable |
| Reformuler | Propose une version humaine, naturelle, non-promotionnelle |
| Orienter | Suggère une Loop autorisée pertinente |
| Résumer | Synthétise le besoin pour les lecteurs de la Boucle |
| Aider à agir | Propose une prochaine action ("Publier cette demande") |
| Suggérer une Loop | Choisit parmi une liste de Loops dont le membre est membre et qui sont autorisées |
| Proposer une prochaine action | Après publication, suggère une étape utile |
| Fallback si confiance faible | Propose une étape de clarification bornée au lieu de forcer un draft |

### L'IA ne peut pas (jamais)

| Interdiction | Raison |
|---|---|
| Publier | La publication est une action métier humaine, pas IA |
| Choisir hors droits serveur | Toute suggestion de Loop doit être revalidée côté serveur |
| Inventer des membres | L'IA ne peut pas suggérer des membres nominativement en MVP |
| Inventer des Loops | L'IA utilise uniquement les Loops fournies par le serveur |
| Devenir un chatbot libre | Pas de conversation prolongée. Une génération = une carte. |
| Remplacer l'humain | L'utilisateur reste décideur à chaque étape |
| Contourner les policies | Les policies Organization + Loop s'appliquent toujours |
| Utiliser OpenAI réel en T074.2 | Aucun appel API externe. FakeAIProvider uniquement. |

## Contrat Produit IA

Issu de T074.1A. Intégré comme contrat fonctionnel stable pour T074.2. Non implémenté ici.

### Champs produit obligatoires

| Champ | Type | Description |
|---|---|---|
| `intent` | enum | `help_request`, `offer`, `question`, `unknown` |
| `confidence` | float 0..1 | Niveau de confiance dans la compréhension |
| `human_reformulation` | string | Reformulation lisible par l'humain |
| `structured_request.title` | string (max 80) | Titre de la demande |
| `structured_request.description` | string (max 800) | Description du besoin |
| `need_type` | string | Type d'aide attendu (conseil, mise en relation, relecture, etc.) |
| `suggested_loop_id` | uuid\|null | Loop suggérée (doit être dans la liste autorisée) |
| `suggested_loop_label` | string\|null | Label lisible de la Loop suggérée |
| `next_action` | object | Action proposée à l'utilisateur |
| `fallback` | object | Fallback si confiance faible |
| `safety` | object | Blocage sécurité si données sensibles |
| `requires_confirmation` | boolean | Toujours `true` |

### Règles de validation

- `requires_confirmation` est toujours `true` — rien n'est jamais automatique
- Si `confidence < 0.65`, `fallback.needed` doit être `true`
- Si `safety.blocked = true`, aucune action de publication n'est affichée
- `suggested_loop_id` doit être validé côté serveur avant affichage
- Le format JSON exact n'est pas imposé pour T074.2, mais les **champs produit sont stables**

### FakeAIProvider

- Support MVP / test fourni par le Lab IA admin (T074.1A)
- Réponses déterministes basées sur 9 scénarios testables
- Aucun coût, aucune dépendance externe
- Provider réel (OpenAI / `laravel/ai`) reporté à T074.7

### Écran 2 — Qui peut m'aider ?

**Image de référence:** `docs/audits/T074.1A-assets/01-qui-peut-maider-reference.png`

**Objectif utilisateur:** Exprimer un besoin flou et obtenir une demande structurée.

**Composants nécessaires:**
- `ConversationInput` — input spacieux, multiligne, bouton d'envoi intégré
- `ExampleFormulations` — exemples de phrases suggérées (optionnel)
- `GeneratingState` — indicateur de génération (subtile, pas de spinner bloquant)
- `ErrorState` — message d'erreur / fallback si la génération échoue

**États UI:**
| État | Comportement |
|---|---|
| Idle | Input conversationnel spacieux + exemples de formulations + bouton d'envoi dans l'input |
| Saisie | L'utilisateur tape son besoin, aucun caractère spécial requis |
| Génération | Animation subtile (pulse ou skeleton), message "BouclePro prépare une proposition..." |
| Erreur | Fallback: "Je n'ai pas pu analyser ta demande. Tu peux reformuler ou décrire ton besoin plus précisément." |
| Fallback confiance faible | Étape de clarification bornée avant de générer |

**Données affichées:**
- Input principal: texte libre
- Exemples de formulations (définies en français): "Je cherche des conseils pour...", "J'ai besoin d'aide avec...", "Je voudrais des retours sur..."
- Aucune bulle chatbot — l'interface reste un input + bouton

**CTA principal:** Bouton d'envoi (icône flèche ou "Analyser") dans l'input

**CTA secondaire:** Revenir à la liste des Boucles

**Comportement mobile:**
- Input prend presque toute la hauteur disponible
- Exemples en chips au-dessus ou en dessous de l'input
- Send button dans l'input, accessible au pouce

**Comportement desktop:**
- Input centré, largeur max maîtrisée (max-w-3xl recommandé)
- Exemples visibles en grille légère
- Touche Entrée (ou Cmd+Entrée) pour envoyer

**Dépendances backend futures:**
- `AiProvider` (interface) — analyse de l'intention
- `FakeAIProvider` — déterministe pour MVP/test
- `AssistedInteractionDraft` DTO — résultat structuré

**Critères d'acceptation:**
- L'input accepte du texte libre multiligne
- Le bouton d'envoi est inactif si l'input est vide
- La génération prend au maximum quelques secondes (même en fallback)
- L'état d'erreur propose toujours une action (reformuler)
- Il n'y a PAS de bulle chatbot
- Il n'y a PAS de conversation à plusieurs tours

**Hors scope bêta:**
- Streaming de tokens
- Reconnaissance vocale
- Suggestions en temps réel pendant la frappe
- Historique des requêtes précédentes

**CTA secondaire:** "Modifier le brouillon" / "Recommencer"

**Comportement mobile:**
- Carte verticale, scroll
- Champs éditables inline
- Boutons en bas, fixés ou non selon la longueur

**Comportement desktop:**
- Carte centrée, largeur max maîtrisée
- Champs éditables avec bordure visible en mode édition
- Aperçu de ce qui sera publié

**Dépendances backend futures:**
- `AssistedInteractionDraft` — stockage temporaire du draft (session ou DB)

**Critères d'acceptation:**
- L'utilisateur peut modifier titre, description et Boucle cible
- Aucune publication n'a lieu sans clic sur "Valider"
- La Boucle conseillée peut être changée (parmi les Loops du membre)
- Le rappel de non-publication est affiché

## Mes Invités Comme Loop System

### Définition

"Mes invités" est une Loop system Organization-scoped liée au parrainage. Elle regroupe les personnes que le membre a parrainées et invitées sur la plateforme.

### Rôle

| Aspect | Valeur |
|---|---|
| Lien avec parrainage | Lié au système de referral existant. Chaque nouvel inscrit invité est rattaché à son parrain via cette Loop. |
| Ce que cela permet en bêta | Voir la liste des personnes invitées, échanger avec elles dans un espace dédié, suivre leur activité. |
| Ce que cela ne doit pas devenir | Pas de CRM agressif, pas de growth hack, pas de classement toxique, pas de notification "ton invité n'a pas été actif depuis 7 jours", pas de pipeline commercial. |

### Règles d'isolation

- **Organization-scoped uniquement** — la Loop "Mes invités" ne peut jamais contenir des personnes hors de l'Organization courante. Aucun cross-tenant.
- **Membres de la même Organization uniquement** — seuls des Members de l'Organization peuvent apparaître.
- **Non supprimable côté parrain** — comme entrée système, elle ne peut pas être supprimée par le parrain.
- **Masquable / mise en sourdine** — pour éviter toute pression relationnelle, le parrain doit pouvoir masquer ou mettre en sourdine cette Loop. Les invités ne sont jamais enfermés dans une relation sociale obligatoire.

### Traitement UI

- "Mes invités" apparaît comme une carte spéciale dans la liste des Boucles
- Elle peut avoir un badge ou un icône distinct (ex: icône de parrainage)
- Le comportement est identique à une Loop normale (messages, demandes d'aide)
- La différence est qu'elle est créée automatiquement et non modifiable par l'utilisateur

### Contraintes

- "Mes invités" n'est pas une Loop comme les autres: elle est créée par le système
- Elle n'a pas de "Quitter la Boucle" — le membre ne peut pas en sortir
- Elle ne compte pas dans les limites de création de Loops
- Elle est masquable / mise en sourdine côté parrain
- Elle peut être officialisée plus tard par OrgAdmin

## Loops Comme Mini-Réseaux de Coordination

### Types de Loops identifiés

| Type | Description | Exemple | Durée |
|---|---|---|---|
| Thématique | Partage d'intérêt ou de compétence | "Graphistes Marseille", "Développement commercial" | Permanente |
| Locale | Liée à un secteur géographique | "Marseille", "Aix-en-Provence" | Permanente |
| Événementielle | Liée à un événement ponctuel | "Afterwork juillet", "Conférence IA" | Temporaire |
| Temporaire | Groupe projet ou mission courte | "Site web V2", "Préparation salon" | Définie |
| Liée à un objectif | Accompagnement d'un but | "Objectif 10 clients" | Définie |
| Officialisable | Loop simple qui devient une structure officielle plus tard | Toute Loop que l'OrgAdmin souhaite reconnaître | Variable |

### Règles

- **Loop ≠ Tenant** — une Loop n'est pas une frontière tenant et n'est pas une isolation base de données
- **Loop reste collaborative** — pas de hiérarchie interne en bêta
- **LoopMember est une frontière d'autorisation applicative** — elle contrôle lecture/écriture/interactions dans une Loop, mais ne remplace pas l'isolation tenant portée par Organization
- **Organization reste la frontière tenant** — l'isolation de sécurité, la séparation des données et le périmètre de billing sont portés par Organization, pas par Loop
- Un membre peut être dans plusieurs Loops
- Les Loops sont toujours internes à une Organization

### Création de Loop (bêta)

- Pas de création par les membres en bêta
- La bêta peut s'appuyer sur des Loops système ou préconfigurées (ex: Loops thématiques activées par défaut)
- "Mes invités" est créé automatiquement
- La création avancée et l'officialisation par OrgAdmin sont reportées à T074.8
- La modération avancée des Loops reste post-bêta

## Reverb-Ready + Fallback Polling

### Principe

T074.2 ne touche pas à la couche temps réel. Mais la spec prépare le terrain.

| Décision | Valeur |
|---|---|
| Reverb installé ? | **Non.** Pas en T074.2. Pas d'installation. |
| Echo configuré ? | **Non.** Pas de package npm Echo. |
| WebSocket en T074.2 ? | **Non.** |
| Fallback polling ? | **Oui.** Acceptable pour le MVP. |
| Événements Reverb-ready ? | Oui — T074.5 et T074.6 doivent préparer des événements broadcast compatibles. |

### Règles de polling (MVP)

- Loop active (ouverte) : polling léger toutes les 5-10 secondes
- Liste de Loops (inactive) : polling toutes les 30-60 secondes
- Badges de notifications : polling en arrière-plan
- Pause ou ralentissement quand l'onglet est caché (Page Visibility API)
- Marquer comme lu : explicite, pas à chaque render

### Critères pour T074.5/T074.6

- Implémentation d'événements `ShouldBroadcast`
- Création de routes de canal (`routes/channels.php`)
- Configuration broadcasting
- Adaptation Echo côté frontend
- Tests de fallback polling avec/sans Reverb

## Non-Objectifs Explicites

Ce que T074.2 ne fait pas et ne prépare pas:

| Non-objectif | Raison |
|---|---|
| **Pas de chatbot libre** | L'IA assiste, ne converse pas. Pas de ChatGPT-like dans BouclePro. |
| **Pas de marketplace Malt** | Pas de mise en relation freelance/client transactionnelle. |
| **Pas de clone Slack/WhatsApp** | Pas de channels bruyants, pas de messagerie instantanée généraliste. |
| **Pas de migration Community → Organization** | `community_id` legacy reste. Migration conceptuelle seulement. |
| **Pas de Reverb** | Pas d'installation. Pas d'Echo. Pas de WebSocket. |
| **Pas d'OpenAI réel** | FakeAIProvider seulement. Pas de clé API. Pas de package provider. |
| **Pas de refonte globale** | Seulement le périmètre ChatLoop + IA-assisted Interactions. |
| **Pas de graph social** | Pas de visualisation des relations entre membres. |
| **Pas de notification agressive** | Notifications calmes, pas de badges partout, pas d'email pour tout. |
| **Pas de gamification toxique** | Pas de classement, pas de points de participation, pas de streaks. |

## Risques et Mitigations

| Risque | Niveau | Mitigation |
|---|---|---|
| Dérive chatbot | Élevé | L'IA ne produit qu'une carte, pas une conversation. `FakeAIProvider` limite les sorties. |
| Bruit conversationnel | Élevé | Polling calme, pas de notification par message, design aéré. |
| Confusion Loop/Tenant | Élevé | Spec explicite: `Loop ≠ Tenant`. Rappelé dans chaque décision. |
| Legacy `community_id` | Moyen | Accepté. Spécifié comme temporaire. Aucune migration. |
| IA qui invente des Loops | Élevé | Toute suggestion serveur-revalidée. Liste autorisée de Loops uniquement. |
| Surcharge cognitive | Moyen | Un seul parcours: intention → clarification → validation → publication. Pas de choix multiples. |
| Publication non validée | Critique | `requires_confirmation = true` toujours. Aucune publication automatique. |
| Coût provider IA | Moyen | `FakeAIProvider` d'abord. Provider réel seulement en T074.7. Quotas par organization. |
| Données sensibles | Élevé | Filtre safety, blocage si données personnelles, contexte minimal envoyé. |
| Reverb prématuré | Moyen | Pas installé. Pas configuré. Polling fallback suffit pour le MVP. |
| Scope T074 trop large | Élevé | Tâches suivantes clairement découpées (T074.3 à T074.11). T074.2 ne fait que la spec. |
