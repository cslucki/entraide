## 1. Executive Verdict

Current messaging is a transaction-thread system, not a general conversation system. It is built around `Transaction -> Message`, buyer/seller roles, transaction status actions, system messages, read state, and email notifications. It should be audited as useful prior art, but not reused as ChatLoop's core model.

Mobile risk is high. The main messages page is a fixed desktop split pane with `w-80` sidebar, `flex` row layout, viewport-height locking, and `overflow-hidden`. The thread mixes transaction controls, review form, message stream, and input in one Livewire-polled column.

Reverb readiness is low. Runtime broadcasting is `log`, there is no `config/broadcasting.php`, no `config/reverb.php`, no `routes/channels.php`, no Echo/Pusher/Reverb frontend setup, and no `ShouldBroadcast` events.

Biggest architectural risk: accidentally building ChatLoop on transaction messaging and inheriting buyer/seller/status coupling, single-recipient read state, and transaction tenant assumptions.

## 3. Current Transaction Messaging Flow

1. Routes: global `/messages` and `/messages/{transaction}` plus scoped `/{community}/messages` and `/{community}/messages/{transaction}` route to `MessageController`.
2. Conversation list: `MessageController::loadConversations()` loads transactions where current user is buyer or seller, eager-loads buyer/seller/service/request/latest message, sorts by `updated_at`.
3. Unread count: `Message` rows are counted where transaction is in the user's loaded transactions, sender is not the user, `read_at` is null, and type is `user`.
4. Open thread: `MessageController::show()` authorizes `view` on the `Transaction`, then renders the same index view with active transaction.
5. Livewire mount: `MessageThread::mount()` receives `Transaction`, stores it, and calls `markRead()`.
6. Render loop: `resources/views/livewire/message-thread.blade.php` uses `wire:poll.3000ms`; each render refreshes transaction, marks messages read, loads all messages ordered by `created_at`.
7. Send: `sendMessage()` validates `newMessage` max 5000, manually checks buyer/seller, blocks `completed/refused/cancelled`, creates `Message`, clears input, touches transaction, marks read, emails recipient.
8. Status actions: transaction approve/refuse/adjust/cancel/complete/confirm/contest live in the message thread UI and post back to `TransactionController`.
9. System messages: `TransactionController` and API transaction controller create `Message` rows with `sender_id = null`, `type = system`.
10. Notifications: new user messages email via `NewMessageReceived`; selected transaction state changes email via `TransactionStatusChanged`.

## 7. Admin / Backoffice Messages Audit

Existing backoffice is `/admin/messages`, backed by `AdminMessageController`. It supports keyword search, sender/buyer/seller search, date filters, detail context of 5 messages before/after, and hard delete.

It is platform-admin global, not OrgAdmin-scoped. Evidence: admin routes only use `auth` + `admin`; controller queries `Message::with(...)` directly with no organization filter. Because `Message` has no tenant scope and admin has no current organization binding, it can see all transaction messages across organizations. That may be acceptable for platform superadmin, but it is not suitable as-is for a future OrgAdmin Message Center.

Future OrgAdmin Message Center needs organization-scoped queries through transaction/loop organization, loop filters, moderation states, audit logs, and probably no hard-delete-by-default.

## 10. Fallback Polling Recommendation

Use fallback polling, but not the current full-thread 3-second render pattern for ChatLoop.

Recommended T074 fallback:

- active open loop thread polls lightweight deltas every 5-10 seconds;
- inactive loop list/unread badges poll every 30-60 seconds;
- pause or slow polling when the tab is hidden;
- fetch only messages after the last seen message ID/timestamp;
- mark-read should be explicit when the thread is visible and stable, not every render.

## 12. Recommended Decisions For T074.2

1. `LoopMessage` should be separate from transaction `Message`.
2. Organization is the tenant; Loop is only a collaborative context.
3. Read state should support multiple members, likely separate from the message row.
4. Realtime should be event-first with polling fallback, not polling-first forever.
5. ChatLoop notifications should be calm: mentions/direct involvement, digest or badge updates, not email for every message by default.
6. Transaction messaging should remain isolated as an Exchange workflow.
7. Admin roles must distinguish platform admin from OrgAdmin.
8. Message moderation should be auditable; avoid hard delete as the primary future action.
