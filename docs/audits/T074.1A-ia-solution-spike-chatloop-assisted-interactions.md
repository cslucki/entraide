## 1. Executive Decision

GO SOUS CONDITIONS.

BouclePro peut introduire une assistance IA pour ChatLoop, mais seulement comme couche de clarification orientee action, pas comme chatbot generaliste. Le MVP doit transformer une phrase floue en proposition d'action publiable, avec validation humaine explicite avant toute publication.

Conditions de GO:

- T074.1A reste un spike documentaire. Aucune IA reelle n'est implementee ici.
- T074.2 doit cadrer le produit: ecrans, vocabulaire, limites UX, confiance, fallback, et relation avec les futures interactions structurees.
- T074.7 doit porter l'implementation IA reelle, apres decision explicite sur provider/package.
- Le MVP initial doit pouvoir fonctionner sans provider externe via un `FakeAIProvider` ou fallback heuristique.
- Les appels IA restent serveur-side, organization-scoped, provider-agnostic, prompt-driven.
- ChatLoop ne doit pas reutiliser le `Message` transactionnel comme coeur metier.
- Organization = Tenant; Loop != Tenant.
- Reverb, packages, composer.json, package.json, API/tenant findings hors scope et T075 restent exclus de cette tache.
- Un Lab IA admin interne est GO SOUS CONDITIONS comme outil de preview design et validation FakeAIProvider uniquement.

Decision produit: l'IA n'est pas un interlocuteur autonome. Elle sert a convertir "J'ai besoin d'aide pour trouver mes premiers clients" en:

- reformulation humaine;
- demande structuree;
- prochaine action;
- Loop pertinente eventuelle;
- bouton "Publier cette demande";
- fallback si confiance faible.

## 3. Recommandation Package Ou No-Package Pour MVP

Recommendation MVP: no-package pour T074.2/T074.7 phase initiale de produit, avec interface interne provider-agnostic et `FakeAIProvider`.

Raisons:

- La valeur MVP est la qualite du contrat produit, pas le choix LLM.
- Les prompts, schemas, seuils de confiance et fallbacks doivent etre stabilises avant d'ajouter une dependance.
- Le projet est Laravel 13 + Livewire 4.2 + Blade/Tailwind, sans package IA installe.
- Installer un starter kit chat importerait une UX contraire a la direction produit.
- Installer `laravel/ai` ou `openai-php/laravel` maintenant brouillerait T074.1A, qui doit rester documentaire.

Decision a reporter a T074.7:

- Si besoin OpenAI simple: `openai-php/laravel`, encapsule derriere `AiProvider`.
- Si besoin multi-provider/structured outputs natifs Laravel: `laravel/ai`, apres verification de compatibilite Laravel 13 et de maturite.
- Ne pas exposer de facade provider dans Livewire, controllers ou Blade.

## 4. Architecture IA MVP Recommandee

Architecture cible pour T074.7, non codee ici:

```text
Livewire ChatLoop composer
    -> AssistedInteractionController ou Action
        -> Authorize organization + loop membership
        -> Build AssistedInteractionInput DTO
        -> ChatLoopPromptBuilder
        -> AiProvider interface
            -> FakeAIProvider en local/test/MVP sec
            -> OpenAiProvider ou LaravelAiProvider plus tard
        -> Validate JSON contract
        -> Confidence gate + safety filters
        -> AssistedInteractionDraft DTO
        -> UI preview card
        -> user clicks "Publier cette demande"
        -> normal domain action creates structured request/interactions
```

Boundaries:

- L'IA propose un brouillon, elle ne publie pas.
- La publication reste une action metier Laravel classique, autorisee et validee.
- Le contexte transmis est minimal: organization id, loop id, user role, phrase utilisateur, categories/loops candidats deja autorises.
- Pas d'historique conversationnel long en MVP.
- Pas de memoire IA persistante en MVP.
- Pas de streaming necessaire pour le MVP; une reponse courte structuree suffit.

Composants recommandes:

| Composant | Responsabilite |
|---|---|
| `AiProvider` | Interface stable: recoit input/prompt, retourne JSON brut ou DTO parse. |
| `FakeAIProvider` | Reponse deterministe pour tests, demo, fallback, cout zero. |
| `ChatLoopPromptBuilder` | Centralise system/user prompts, vocabulaire BouclePro, contraintes JSON. |
| `AssistedInteractionInput` | Texte utilisateur + contexte autorise. |
| `AssistedInteractionDraft` | Reponse structuree validee avant affichage. |
| `AssistedInteractionPolicy` | Verifie organization + loop membership + droits de publier. |
| `AssistedInteractionLogger` | Journal minimal sans secret ni prompt interne complet en clair. |

## 5. Contrat JSON Minimal

Contrat recommande pour une premiere iteration:

```json
{
  "intent": "help_request",
  "confidence": 0.82,
  "human_reformulation": "Tu veux trouver tes premiers clients et tu cherches des retours concrets pour avancer.",
  "structured_request": {
    "title": "Trouver mes premiers clients",
    "description": "Je cherche des conseils, retours d'experience ou contacts pour identifier mes premiers clients et lancer mes premieres demarches.",
    "need_type": "advice",
    "audience": "membres pouvant partager une experience commerciale",
    "suggested_loop_id": "uuid-or-null",
    "suggested_loop_label": "Developpement commercial"
  },
  "next_action": {
    "label": "Publier cette demande",
    "action": "publish_help_request",
    "requires_confirmation": true
  },
  "fallback": {
    "needed": false,
    "message": null,
    "questions": []
  },
  "safety": {
    "contains_sensitive_data": false,
    "needs_human_review": false,
    "blocked": false
  }
}
```

Validation minimale:

- `intent`: enum limitee (`help_request`, `offer`, `question`, `unknown`).
- `confidence`: nombre entre 0 et 1.
- `human_reformulation`: texte court, francais, non promotionnel.
- `structured_request.title`: 80 caracteres max.
- `structured_request.description`: 800 caracteres max.
- `suggested_loop_id`: null ou id issu d'une liste preautorisee par le serveur.
- `next_action.requires_confirmation`: toujours `true`.
- Si `confidence < 0.65`, `fallback.needed` doit etre `true`.
- Si `safety.blocked = true`, aucune action de publication n'est affichee.

## 6. Prompts Systeme / Utilisateur Proposes

System prompt propose:

```text
Tu aides les membres de BouclePro a transformer une phrase floue en prochaine action claire.
Tu n'es pas un chatbot et tu ne dois pas prolonger une conversation artificielle.
Tu dois proposer un brouillon court, humain, utile et publiable.
Tu respectes strictement ces regles:
- reponds uniquement en JSON valide conforme au schema demande;
- ecris en francais naturel;
- ne promets pas de resultat;
- ne donne pas de conseil juridique, medical, financier ou sensible;
- ne publie rien toi-meme;
- si l'intention est ambigue, demande au maximum deux clarifications;
- Organization est le tenant; Loop est un contexte collaboratif, jamais une frontiere de securite;
- n'utilise que les loops candidates fournies par le serveur;
- n'invente pas de donnees personnelles, de membres, de loops ou de permissions.
```

User prompt template propose:

```text
Contexte autorise:
- organization_id: {{ organization_id }}
- loop_id_actuel: {{ current_loop_id_or_null }}
- loops_candidates: {{ authorized_loop_candidates_json }}
- user_role: {{ organization_role }}
- locale: fr-FR

Phrase utilisateur:
{{ user_input }}

Produit attendu:
Transforme cette phrase en brouillon d'action publiable.
Retourne uniquement le JSON conforme au contrat.
```

Prompt hardening:

- Ne jamais envoyer tous les messages du Loop.
- Ne jamais envoyer les prompts internes visibles en UI.
- Ne jamais envoyer de secrets, API keys, configuration admin, ou donnees cross-organization.
- Ajouter un filtre serveur avant appel IA pour donnees sensibles manifestes.
- Revalider toute suggestion de Loop cote serveur.

## 7. Fallback Sans IA

Fallback no-IA recommande:

1. Detecter quelques intentions simples par boutons ou heuristiques locales:
   - "J'ai besoin d'aide..." -> `help_request`
   - "Je propose..." -> `offer`
   - "Je cherche..." -> `help_request`
   - inconnu -> clarification courte
2. Generer un brouillon conservateur:
   - titre depuis les premiers mots utiles;
   - description = phrase utilisateur nettoyee;
   - Loop = Loop courant si l'utilisateur y est membre, sinon null;
   - confiance = 0.5;
   - fallback.needed = true si trop court ou ambigu.
3. Afficher une carte editable:
   - reformulation;
   - champs titre/description;
   - select Loop autorisee;
   - bouton "Publier cette demande";
   - lien "Modifier avant publication".

## 8. Risques Cout / Securite / Confidentialite

| Risque | Niveau | Mitigation MVP |
|---|---|---|
| Derive vers chatbot | Eleve | UI en carte d'action, pas en assistant conversationnel. Une generation, puis validation humaine. |
| Cout non maitrise | Moyen | No-package/FakeAIProvider d'abord, quotas par organization, input court, pas de streaming en MVP. |
| Donnees confidentielles envoyees au provider | Eleve | Minimize context, redaction simple, avertissement UI, server-side only, logging prudent. |
| Fuite cross-organization | Eleve | Contexte construit depuis organization courante; Loop candidates autorisees seulement; validation serveur du `suggested_loop_id`. |
| Prompt injection | Moyen | JSON schema strict, allowlist actions, aucune execution d'outils, publication manuelle uniquement. |
| Hallucination de Loop/membre | Moyen | Le modele ne peut choisir que parmi les ids fournis; toute suggestion inconnue est rejetee. |
| Conseils sensibles | Moyen | Safety gate + refus/fallback pour juridique/medical/financier/confidentiel. |
| Dependence provider | Moyen | Interface provider-agnostic + FakeAIProvider + config centrale. |
| UX confuse | Eleve | Copier T074.2 avant code: l'IA doit clarifier, pas discuter. |

## 9. Limites MVP

Le MVP ne doit pas inclure:

- chat IA libre;
- memoire IA;
- embeddings;
- recherche semantique;
- tool calling;
- auto-publication;
- moderation IA;
- streaming obligatoire;
- Reverb;
- suggestions de membres nominatives;
- entrainement sur messages de Loop;
- historique conversationnel cross-loop;
- admin AI center.

Le MVP doit se limiter a:

- une phrase utilisateur;
- une proposition structuree;
- un fallback;
- une validation humaine;
- une publication via workflow metier normal.

### Scenarios FakeAIProvider A Prevoir

| Scenario | Input admin | Sortie attendue |
|---|---|---|
| Besoin client clair | "J'ai besoin d'aide pour trouver mes premiers clients." | Confiance haute, demande clarifiee, Boucle conseillee commerciale, message_draft publiable. |
| Besoin trop vague | "Je suis bloque." | Confiance faible, fallback actif, 1 ou 2 questions courtes, pas de Boucle conseillee ferme. |
| Demande avec deadline | "Je dois trouver quelqu'un pour relire mon offre avant vendredi." | Deadline detectee, type d'aide relecture/conseil, ton direct, validation humaine. |
| Mauvais canal | "Je veux vendre mon service a tout le monde." | Reformulation prudente, ton non promotionnel, rappel d'entraide, pas d'auto-publication. |
| Donnees sensibles | "Voici le numero perso d'un prospect..." | Safety `contains_sensitive_data=true`, preview bloquee ou fortement avertie. |
| Loop ambigue | "Je cherche des avis sur mon site et ma strategie." | Plusieurs interpretations possibles, fallback ou Boucle conseillee null avec question. |
| Intention offre | "Je peux aider a refaire une page de vente." | Intent `offer`, message_draft oriente proposition d'aide, pas `help_request`. |
| Hors scope | "Donne-moi une strategie juridique pour mon contrat." | Safety review/blocked, fallback vers clarification ou refus doux. |

Ces scenarios doivent etre deterministes. Le meme input doit produire la meme preview afin de comparer les maquettes, tests Playwright et decisions produit.

Solution sans package, sans migration, sans env obligatoire. Utilise `APP_ENV` (valeur par defaut Laravel).
