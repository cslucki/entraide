## Architecture Actuelle

### Middleware Tenant

```
Route {community} ===[ResolveCommunity]===>
    app()->instance('current_community', $community)
    app()->instance('current_organization', $community)
    View::share('currentCommunity', $community)
    View::share('currentOrganization', $community)
```

**Résolution Runtime :**
```
CurrentOrganization::get()
    1. app('current_organization')    ← préféré
    2. app('current_community')       ← fallback legacy
    3. null                           ← aucun tenant
```

**BelongsToTenantScope :**
```
Si CurrentOrganization::get() !== null
    → WHERE {table}.community_id = {org->id}
Si null
    → AUCUNE clause WHERE ajoutée (scope silencieux)
```

## Notes Architecturales

1. **Organization = Community (alias)** : Organization étend Community avec `$table = 'communities'`. C'est intentionnel et correct pour la couche de compatibilité.

2. **community_id = organization_id** : Les deux colonnes existent. `HasOrganizationId` trait sync auto. `BelongsToTenantScope` filtre par `community_id`. C'est correct et stable.

3. **current_organization = current_community** : Les deux bindings pointent vers la même instance Community. `CurrentOrganization::get()` préfère `current_organization`. C'est le pattern cible.

4. **Routes `/org/{organization}` préparées mais commentées** : Middleware `organization` alias existe, $organizationConstraint défini, mais le prefix n'est pas activé. Prochaine étape logique.
