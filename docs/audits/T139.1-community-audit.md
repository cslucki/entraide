## Résumé exécutif

Cet audit couvre le périmètre suivant du codebase pour mesurer l'ampleur de la dette technique liée au concept **Community** et préparer le programme de migration vers **Organization** :

**Périmètre audité :** `app/`, `tests/`, `database/`, `routes/`, `resources/`, `config/`, `public/`
**Exclus de la baseline de progression :** `docs/`, `TODO/`, `_temp/`, historiques git, `ai/` (hors audits)

> **Note :** Les fichiers `docs/`, `TODO/` et `ai/` sont exclus car ils contiennent la documentation de la migration elle-même. Leur contenu Community reflète l'état de la migration, pas la dette technique résiduelle.

**Architecture clé identifiée :** `Organization extends Community` avec `$table = 'communities'`. Les deux modèles pointent vers la **même table** `communities`. Il n'existe pas de table `organizations` séparée.

### 2.5 Risques DB

1. ⚠️ **organization_id sans table cible** — 7 colonnes existent sans FK. Aucune protection d'intégrité référentielle.
2. ⚠️ **Loops n'a PAS d'organization_id** — Seule table avec `community_id` qui n'a pas reçu la migration d'ajout.
3. ⚠️ **BelongsToTenantScope filtre par `community_id`** — Si les colonnes divergent, le scope ne protège que sur `community_id`.
4. ⚠️ **User, BlogPost, Loop sans BelongsToTenantScope** — Isolation repose uniquement sur le code applicatif.
5. ⚠️ **Migration `2026_05_12_101622` DOWN protégée SQLite** — Impossible de rollback sur SQLite.

## 3. Routes & Middleware

### 3.1 Architecture des routes

| Groupe | Middleware | Dépendance Community |
|--------|-----------|---------------------|
| `/{community}/...` (L277-374) | `['web', 'community']` | **P0** — coeur de la plateforme |
| `admin/communities/*` | admin | **P0** — CRUD direct sur Community |
| `admin/meta_community` | admin | **P0** — meta-communauté |
| `admin/users/{user}/assign-community` | admin | **P0** — assignation community_id |
| `partenaires/demande` | web | **P2** — CommunityRequest |
| API (`routes/api.php`) | `api.organization` | **P0** — via `$user->community_id` |
| Broadcast (`channels.php`) | auth | **P0** — comparaison `community_id` |

### 3.2 Middleware critiques

| Middleware | Rôle | Dépendance |
|-----------|------|-----------|
| `ResolveCommunity` | Résout tenant par slug | Route `{community}` |
| `ResolveUrlOrganization` | Résolution auto routes non-prefixées | `$user->community_id` + `Community::find()` |
| `ResolveApiOrganization` | Résolution API | `$user->community_id` |
| `BelongsToTenantScope` | Global scope | `current_community`/`current_organization` |

### 3.3 Routes nommées `community.*`

Toutes les routes prefixées `/{community}/...` reçoivent le préfixe de nom `community.` (30+ routes : `community.home`, `community.login`, `community.dashboard`, etc.).

### 3.4 Recommandations routes

1. Créer un groupe `/{organization}/...` avec l'alias 'organization' (déjà prêt dans `ResolveOrganization extends ResolveCommunity`)
2. Déprécier les routes `/{community}` (garder en redirect)
3. Migrer les route names de `community.*` → `organization.*`
4. Planifier la migration des URLs

## 7. Recommandations par ordre de traitement

### Phase 1 — Gates & caractérisation (T139.2 — en cours)

Avant toute migration, établir des gates de sécurité :
1. **Smoke tests** sur toutes les routes critiques GET (/explorer, /membres, /blog, /boucles, /dashboard, /admin/dashboard)
2. **Tests de caractérisation** du comportement actuel (BelongsToTenantScope sur community_id, CurrentOrganization fallback, ResolveCommunity bindings, routes /{community}, Loop community_id, broadcast channels)
3. **Known-risk tests** documentés, non bloquants, marqués `@group tenant-known-risk`
4. Script smoke local optionnel

> ⚠️ **Important :** `current_community` n'est pas à supprimer maintenant. Il est à couvrir par des gates de caractérisation. La suppression sera traitée dans T140.3 après migration des callers.

> ⚠️ **Important :** Les routes `/{community}` sont une dette legacy compatible. Elles ne doivent pas être renommées maintenant. La stratégie future est de créer `/org/{organization}` en parallèle, puis redirects, puis dépréciation.

> ⚠️ **Important :** `Organization extends Community` est viable comme couche temporaire V1 tant que les gates restent vertes (car Organization utilise la même table `communities` avec `$table = 'communities'`). Une vraie table `organizations` séparée serait une migration dédiée future (T140.9), pas un prérequis de T139.2.

### Phase 2 — Isolation tenant (T140.1 → T140.2)

1. Migrer `BelongsToTenantScope` pour utiliser `organization_id` (nécessite colonne déjà présente)
2. Ajouter `organization_id` à la table `loops` + adapter `LoopFactory`
3. Ajouter `HasOrganizationId` trait sur `Loop`

### Phase 3 — Routage parallèle (T140.4)

1. Créer le groupe `/{organization}/...` (alias middleware 'organization' déjà prêt)
2. Garder les routes `/{community}` en redirect compat
3. Renommer les route names `community.*` → `organization.*`
4. Mettre à jour les helpers E2E

### Phase 4 — current_community fallback (T140.3)

1. Couvrir les derniers callers par des gates
2. Supprimer le fallback `current_community` dans `CurrentOrganization::get()`
3. Unifier les bindings : ne binder que `current_organization`

### Phase 5 — Nettoyage modèles et contrôleurs (T140.5, T140.7)

1. Dans `HasOrganizationId`, faire d'`organization_id` la source unique
2. Migrer les contrôleurs (requêtes `community_id` → `organization_id`)
3. Supprimer les colonnes `community_id` (post-validation)

### Phase 6 — Tests (T140.6)

1. Corriger `ExplorerTest.php` (9 tests faux positifs) — cleanup après stabilisation
2. Migrer les tests Loop vers `Organization::factory()`
3. Ajouter tests `WithoutGlobalScopes`
4. Ajouter tests E2E isolation cross-organization
