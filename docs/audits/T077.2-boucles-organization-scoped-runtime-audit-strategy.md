# T077.2 — Boucles Organization-Scoped Runtime Audit & Strategy

**Date :** 2026-05-18 19:42 Europe/Paris  
**Auditeur :** OPENCODE  
**Statut :** COMPLET  
**Mode :** Audit / stratégie uniquement, aucun code runtime modifié

---

## Résumé Exécutif

L'existant Loop est fonctionnel pour des usages authentifiés internes, mais il n'est pas encore prêt pour une exposition runtime DB publique ou semi-publique sur `/boucles`.

Les protections actuelles reposent surtout sur des garde-fous applicatifs dans `LoopController`, `LoopService` et `LoopMessageService` : filtrage par tenant legacy `community_id`, vérification d'appartenance utilisateur, et vérification membership actif. Ces protections sont utiles mais restent incomplètes pour T077.3 parce que le modèle `Loop` ne possède pas encore de colonne canonique `organization_id`, aucun `LoopPolicy` n'existe, et le route-model binding `{loop}` charge d'abord le record par UUID avant que le controller ne rejette les accès hors tenant.

Décision d'audit : T077.3 doit viser un MVP ciblé de visibilité et membership Organization-scopé, sans nouvelle exposition publique DB large. La route publique `/boucles` doit rester contrôlée tant qu'un contrat de visibilité explicite n'existe pas.

---

## Règles Architecture Retenues

| Règle | Application à T077.3 |
|---|---|
| Organization = Tenant | Toute lecture runtime Loop doit être bornée par l'Organization résolue. |
| Loop ≠ Tenant | Une Loop est un espace collaboratif interne à une Organization, pas une frontière de sécurité autonome. |
| Partner ≠ Tenant | Aucune logique Loop ne doit être scopée par Partner. |
| Public ≠ global | Une surface publique peut exister, mais elle doit résoudre une Organization avant toute lecture DB. |
| `current_organization` canonique | Les nouvelles décisions runtime doivent partir de `current_organization`, pas du fallback legacy. |
| `organization_id` canonique côté code | T077.3 doit préparer l'usage d'un identifiant Organization explicite même si la DB Loop reste temporairement sur `community_id`. |
| `community_id` legacy DB temporaire | Acceptable uniquement comme stockage existant documenté. |

## État Actuel des Modèles Loop

### `Loop`

| Élément | État |
|---|---|
| Tenant storage | `community_id` obligatoire, FK vers `communities.id`. Legacy DB temporaire. |
| Identifiant canonique code | Pas de champ `organization_id` sur `loops`. |
| Relations | `community()`, `creator()`, `members()`, `activeMembers()`, `messages()`. |
| Unicité | `unique(['community_id', 'slug'])`. |
| Visibilité | Aucun champ de visibilité publique ou interne explicite. |
| Statut | `status` string, défaut `active`, `archived` utilisé en factory/test. |
| Type | `type` string, défaut `custom`, valeurs vues : `custom`, `system`, `team`, `project`, `social`. |

Conclusion : le modèle ne peut pas encore exprimer proprement une visibilité T077.3. Toute exposition DB doit donc être limitée à des règles dérivées du tenant et du membership, ou attendre un champ de visibilité explicite dans une tâche ultérieure.

### `LoopMember`

| Élément | État |
|---|---|
| Storage | `loop_id`, `user_id`, `role`, `status`, `joined_at`. |
| Unicité | `unique(['loop_id', 'user_id'])`. |
| Rôles | `owner`, `moderator`, `member` dans factories/services. |
| Statuts | `active`, `invited`, `left` dans factories/tests. |
| Tenant direct | Aucun `organization_id` / `community_id`; le tenant est indirect via `loop.community_id` et `user.community_id`. |

Conclusion : membership réel existe déjà et doit rester la base d'accès à `show`, messages et mutations. T077.3 doit formaliser les droits autour de ce modèle au lieu d'exposer les Loops uniquement par statut.

### `LoopMessage`

| Élément | État |
|---|---|
| Storage | `loop_id`, `sender_id`, `body`, `type`, `metadata`. |
| Types | `user`, `help_request`, `system` en factory/event/tests. |
| Tenant direct | Aucun tenant direct; dépend du parent `Loop`. |
| Broadcast | `LoopMessageCreated` diffuse sur private channel `loop.{loopId}`. |

Conclusion : les messages ne doivent jamais être exposés sur une surface publique T077.3. Ils restent strictement membres actifs.

## État Actuel des Routes `/loops` et `/boucles`

### `/boucles`

| Route | Controller | Middleware | État |
|---|---|---|---|
| `GET /boucles` | `HomeController::boucles` | public web | Surface publique marketing, ne doit pas lire les vrais records Loop aujourd'hui. |
| `GET /boucles/creer` | redirect vers `/partenaires/demande` | public web | Route historique hors objectif T077.3, à ne pas utiliser pour runtime Loop. |

`/boucles` est actuellement la seule URL publique francophone visible pour les Boucles. Suite à T077.1, elle doit rester une surface contrôlée et ne pas exposer `Loop::query()` tant que T077.3 n'a pas défini une visibilité Organization-scopée.

### `/loops` root domain authentifié

| Route | Controller | Protection actuelle |
|---|---|---|
| `GET /loops` | `LoopController::index` | `auth`; filtre `Loop.community_id = resolved tenant id`; ne montre que les loops où l'utilisateur est membre. |
| `GET /loops/create` | `LoopController::create` | `auth`; exige tenant résolu et appartenance utilisateur. |
| `POST /loops` | `LoopController::store` | `auth`; crée via `LoopService::createLoop($request->user())`. |
| `GET /loops/{loop}` | `LoopController::show` | `auth`; binding par UUID puis checks tenant + membership actif. |
| `POST /loops/{loop}/members` | `LoopController::addMember` | `auth`; tenant + membership owner/moderator + referral. |
| `POST /loops/{loop}/messages` | `LoopController::storeMessage` | `auth`; tenant check controller + `LoopMessageService::assertCanSend`. |
| `POST /loops/{loop}/help-request/*` | `LoopController` | `auth`; tenant + membership; inclut IA existante, hors scope T077.3. |

Risque root-domain : en absence de tenant URL, le controller retombe sur le tenant du profil utilisateur. Cela évite une lecture globale mais ne matérialise pas `Public ≠ global` par URL. Pour T077.3, toute exposition publique doit résoudre une Organization avant lecture.

### Routes préfixées Organization legacy

Les routes `/{community}/loops/*` existent sous middleware `community`. Ce middleware bind aussi `current_organization` aujourd'hui, mais le nom de route et le segment restent legacy. Ces routes sont utiles pour audit de comportement tenant, pas comme nouveau contrat lexical à étendre.

### Admin

`GET /admin/loops` utilise `AdminLoopController::index` et filtre `Loop::where('community_id', $user->organization_id ?? $user->community_id)`. Cette surface est admin-authentifiée mais ne passe pas par une policy Loop dédiée.

## Risques Cross-Organization

| Risque | Sévérité | Détail | Mitigation recommandée |
|---|---|---|---|
| Route-model binding non scoped | P1 | `{loop}` charge par UUID avant check controller. Le controller rejette ensuite, mais le binding lui-même n'est pas tenant-scopé. | T077.3 doit ajouter des tests de non-exposition et envisager binding scoped ou lookup contrôlé par Organization dans les endpoints concernés. |
| Root-domain tenantless | P1 | `/loops` utilise fallback profil utilisateur sans tenant URL; `/boucles` public ne doit pas en déduire une lecture globale. | Pour `/boucles`, résoudre une Organization explicite avant toute lecture DB. Fail closed si aucune Organization. |
| Absence de visibilité explicite | P1 | `status = active` ne signifie pas public. | Ne jamais utiliser seulement `status` comme filtre public. Définir le contrat de visibilité dans T077.3. |
| Pas de policy Loop | P2 | Autorisation dispersée, difficile à réutiliser pour T077.3. | Ajouter une policy ou un gate minimal dans T077.3 seulement si nécessaire, sans refactor lourd. |
| Messages dépendants du parent | P1 | `loop_messages` n'a pas de tenant direct; toute fuite Loop entraîne fuite messages si chargé. | Interdire les messages sur public T077.3. Charger uniquement des compteurs sûrs si besoin. |
| Broadcast channel | P2 | `Loop::find()` global puis checks. Pas dans le scope T077.3. | Ne pas toucher websocket; garder hors scope. |
| Admin fallback | P2 | `organization_id ?? community_id` peut masquer un profil admin incomplet. | Tester si T077.3 touche admin; sinon documenter. |

## Risques d'Exposition Publique Prématurée

Ne pas brancher `/boucles` sur `Loop::where('status', 'active')`.

Pourquoi :

- `active` exprime l'état interne, pas la visibilité publique.
- `loops` n'a pas de champ `organization_id` canonique ni `visibility`.
- Une Loop peut contenir des noms, descriptions ou membres sensibles pour une Organization.
- Les messages et help requests sont explicitement collaboratifs et internes.
- Root domain sans Organization résolue deviendrait global par accident.

## Stratégie Recommandée pour T077.3

### Objectif

Construire un MVP `Boucles Visibility & Membership` ciblé, Organization-scopé, sans ChatLoop, sans IA, sans websocket, sans exposition globale.

### Séquence proposée

1. Définir le contrat de visibilité minimal : `private/internal` par défaut, et une visibilité publique uniquement si une donnée existante fiable le permet ou si une tâche migration dédiée l'ajoute plus tard.
2. Garder `/boucles` sans lecture DB tant que la visibilité explicite n'existe pas, ou faire une lecture fail-closed uniquement après résolution d'Organization.
3. Pour les routes authentifiées, conserver membership actif comme condition de `show` et mutations.
4. Pour une éventuelle liste publique T077.3, exposer uniquement `name`, `slug`, `description` courte, `type`, compteur agrégé sûr; jamais messages, membres individuels, referrals ou créateur email.
5. Ajouter des tests avant modification runtime : visiteur root-domain sans tenant ne voit aucun record DB; visiteur Organization-scopé ne voit que les Loops explicitement visibles; non-membre même tenant ne voit pas `show`; cross-Organization impossible par UUID.
6. Si création/modification de policy nécessaire, rester minimal : `viewAny`, `view`, `manageMembers`, `sendMessage`, avec Organization résolue et membership.
7. Reporter tout renommage large `Community` vers une tâche migration dédiée; ne pas faire de search/replace.

### Point d'attention

La DB actuelle ne permet pas de différencier proprement une Loop interne active d'une Loop publiquement listable. Si T077.3 doit réellement exposer des records DB publiquement, il faudra soit limiter l'exposition à des fixtures contrôlées, soit prévoir une tâche de migration dédiée pour un champ de visibilité. Sans cela, l'option sûre est de garder `/boucles` statique et de concentrer T077.3 sur membership authentifié.

## Critères d'Acceptation Proposés pour T077.3

- `/boucles` ne lit jamais les Loops DB sans Organization résolue.
- Aucune donnée Loop cross-Organization n'est visible sur root domain ou route préfixée.
- `status = active` seul n'est jamais utilisé comme définition de visibilité publique.
- Un visiteur non connecté ne voit aucun message, aucun membre individuel et aucune donnée privée de Loop.
- Un membre connecté voit uniquement ses Loops internes via membership actif.
- Un non-membre même Organization ne peut pas accéder à `loops.show` interne.
- Un Organization Admin ne voit que les Loops de son Organization.
- Le route-model binding `{loop}` est couvert par test cross-Organization par UUID.
- Les tests existants `Loop*`, `AdminLoopsTest`, `T07411RoutesTenantSafetyTest`, `PublicFrenchPartnersRoutesTest` restent verts.
- Aucun ChatLoop, IA, websocket, migration lourde ou refactor global n'est introduit.

## Conclusion

T077.3 peut avancer, mais uniquement en fail-closed : Organization résolue obligatoire avant toute lecture DB publique, membership actif obligatoire pour l'espace interne, et aucune exposition de messages ou de roster sur public. La dette legacy `community_id` reste acceptable comme stockage temporaire documenté, mais ne doit pas devenir le vocabulaire normal des nouvelles surfaces.
