# T140.4 — Routes /org/{organization} en parallèle

## Résumé

Ajout d'un groupe de routes `/org/{organization}` en parallèle des routes legacy
`/{community}`, sans redirect, sans suppression, sans modification DB.

## Stratégie parallèle

- Mêmes controllers que les routes legacy `/{community}`
- Mêmes middlewares : `web` group + `organization` alias (ResolveOrganization)
- Même contrainte regex de slug
- Aucun redirect
- Aucune suppression

## Compatibilité legacy

- Toutes les routes `/{community}` et `community.*` sont conservées à l'identique
- Les bookmarks existants continuent de fonctionner
- Les URLs générées par `route('community.home', ...)` dans les controllers/views ne changent pas
- Les tests legacy characterization ne sont pas modifiés

## Patch middleware

`app/Http/Middleware/ResolveUrlOrganization.php` — ligne 119 :

```php
// Avant :
return $route && $route->hasParameter('community');

// Après :
return $route && ($route->hasParameter('community') || $route->hasParameter('organization'));
```

**Problème résolu :** `ResolveUrlOrganization::isCommunityPrefixedRoute()` ne vérifiait
que le paramètre `community`. Sur les routes `/org/{organization}`, le middleware
tentait de résoudre l'organisation depuis l'utilisateur authentifié, pouvant binder
une organisation DIFFÉRENTE de celle dans l'URL. La correction ajoute la vérification
du paramètre `organization`, ce qui fait skipper ResolveUrlOrganization pour laisser
ResolveCommunity/ResolveOrganization gérer la résolution.

**Ordre de résolution désormais correct :**
1. `ResolveUrlOrganization` (groupe web) → SKIP (car route a `organization` param)
2. `ResolveOrganization` alias de `ResolveCommunity` (groupe route) → resolve par slug
3. `current_organization` et `current_community` bindés correctement
