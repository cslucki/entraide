# T075.10 — Community Legacy Code Audit & Removal Plan

## 1. Executive Summary

This document is the authoritative audit and removal plan for all remaining Community legacy code in the BouclePro codebase.

**Total legacy references found:** ~2,000+ across ~150+ files
**Core model:** `app/Models/Community.php` (Organization extends Community)
**Critical middleware:** `app/Http/Middleware/ResolveCommunity.php`
**Runtime bindings:** `current_community`, `$currentCommunity`
**DB columns:** `community_id` on 8+ tables (users, services, service_requests, transactions, blog_posts, loops, referrals, referral_rewards)
**Scope:** `app/Models/Scopes/BelongsToTenantScope.php` (scopes on community_id)
**Route prefix:** `/{community}` group with `community.*` named routes
**E2E tests:** Full `tests/e2e/community-transactions/` helper module and 9+ spec files

**Key conclusion:** Most Community debt is **safe to keep temporarily** because the Organization migration already created an intermediate compatibility layer. The `organization_id` column exists alongside `community_id`, `Organization extends Community`, and `ResolveOrganization extends ResolveCommunity`. The runtime resolution falls back: `current_organization` → `current_community` → null.

**T075.10 ne déclenche aucune correction.** Ce rapport est un audit et un plan. Chaque correction future listée ici doit devenir une tâche séparée, une branche, un TASK file.

**Priority breakdown:**
- **P0 (risque critique à séquencer dans une tâche dédiée) :** 2 items — `BelongsToTenantScope` source de vérité tenant à clarifier/tester, `HasOrganizationId` stratégie Organization-first à spécifier
- **P1 (dette conceptuelle/haute à traiter dans des tâches séparées après P0) :** ~10 items — ResolveOrganization implémentation réelle ou suppression, ResolveCommunity dépréciation, services/controllers dépendant de `$user->community_id`
- **P2 (dette acceptable, pas d'impact sécurité immédiat) :** ~50+ items — docs, migrations, seeders, factories, tests historiques, `{community}` route param, `community.*` route names, vues legacy, `$currentCommunity` view variable

## 2. Architecture Context & Rules

### Canonical Rules

| Concept | Is Tenant? | Role |
|---------|-----------|------|
| **Organization** | YES | Security, billing, governance boundary |
| **Loop** | NO | Collaborative group within an Organization |
| **Partner** | NO | Co-branding/distribution entry |
| **Community** | LEGACY | Temporary backward compatibility layer |

### Runtime Resolution Chain (Canonical)

```php
$organization = app()->bound('current_organization')
    ? app('current_organization')
    : (app()->bound('current_community')
        ? app('current_community')
        : null);
```

### Class Hierarchy

```
Community (legacy base model)
  └── Organization extends Community (current canonical model)

ResolveCommunity (legacy middleware)
  └── ResolveOrganization extends ResolveCommunity (empty — used for clarity)
```

### View Variable Fallback

```blade
$tenant = $currentCommunity ?? $currentOrganization ?? null;
```

This fallback is used in `navigation.blade.php`, `dashboard.blade.php`, and `layouts/app.blade.php`.

### Key Observation

The migration strategy already created **dual runtime compatibility**. Both `current_organization` and `current_community` are bound simultaneously. The `HasOrganizationId` trait syncs `organization_id = community_id` on creating/updating. This means the system works correctly today, but the Community layer is dead weight.

## 6. Dependency & Coupling Map

### 6.1 Critical Dependency Chain

```
communities DB table
  └── Organization extends Community (class inheritance)
      ├── ResolveUrlOrganization queries Community::find()
      ├── ResolveApiOrganization reads $user->community_id
      ├── HasOrganizationId syncs community_id → organization_id
      └── BelongsToTenantScope scopes on community_id
           └── Models using HasOrganizationId + BelongsToTenantScope
                ├── User (community_id fillable, community() relationship)
                ├── Service (community_id + organization_id)
                ├── ServiceRequest (community_id + organization_id)
                ├── Transaction (community_id + organization_id)
                ├── BlogPost (community_id + organization_id)
                ├── Loop (community_id NOT NULL + unique(community_id, slug))
                ├── Referral (community_id + organization_id)
                └── ReferralReward (community_id + organization_id)

ResolveCommunity middleware
  └── bootstrap/app.php 'community' alias
      └── routes/web.php /{community} group (community middleware)
          ├── Controllers receiving resolved tenant
          │   └── Services querying on community_id
          └── Views with $currentCommunity fallback

CommunityFactory
  └── OrganizationFactory extends CommunityFactory
      └── LoopFactory uses Community::factory()
          └── Tests use Community::factory()
```

### 6.2 What Breaks If Each Component Is Removed

| Component Removed | Immediate Failure |
|-------------------|-------------------|
| `Community` model | All `Organization extends Community` breaks — entire app fails |
| `ResolveCommunity` middleware | All routes using `'community'` middleware alias fail |
| `BelongsToTenantScope` | No tenant scoping — data leak to all users |
| `HasOrganizationId` | `organization_id` stops syncing — data drift |
| `community_id` column | All Eloquent queries filtering on `community_id` fail |
| `current_community` binding | Fallback chain breaks — code using `app('current_community')` fails |

### 6.3 What Is Already Decoupled

- **ResolveOrganization** already exists as the canonical replacement for `ResolveCommunity`
- **current_organization** binding already exists alongside `current_community`
- **organization_id** column already exists alongside `community_id` on all business tables
- **ResolveUrlOrganization** already handles the full resolution pipeline
