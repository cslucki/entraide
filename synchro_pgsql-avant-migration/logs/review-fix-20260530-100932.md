## Risques restants

1. **PHP ETL non testé** : le self-test ne lance pas l'ETL. Le premier vrai dump révélera d'éventuels problèmes de mapping de colonnes.
2. **`pg_restore` dans la DB temporaire** : utilisation de l'user `postgres` — si la DB PROD a des extensions ou types non installés localement, le restore peut échouer.
3. **Blacklist** : certaines tables PROD pourraient avoir des colonnes FK pointant vers des tables blacklistées (ex: `personal_access_tokens` référencé par `users`). Le `ON CONFLICT DO NOTHING` gère les doublons, mais pas les contraintes FK.
4. **Permission `LOCAL_PG_ADMIN_PASSWORD`** : passée en variable d'environnement — peut apparaître dans les logs shell si mal gérée.
5. **Org `main`** : doit exister localement avant le premier lancement — le script échoue clairement si absente.
6. **Pas de dry-run PROD dump** : le premier dump PROD est un vrai dump, pas un --dry-run. C'est voulu (read-only), mais ça consomme une connexion PROD.
