# Preflight RAA Findings

## Fresh-install schema chain (current)

Migration order on `migrate:fresh`:

1. `2026_05_13_000001_create_referrals_table.php`
   - Creates `referrals` with **both** `community_id` (nullable) + `organization_id` (nullable)
   - FK: `community_id` → `organizations`
   - Unique: `[community_id, referrer_user_id, referred_user_id]`
   - ⚠️ `drop_community_id_from_tables` up() drops community_id unconditionally (no hasColumn guard)

2. `2026_05_13_000002_create_referral_rewards_table.php`
   - Creates `referral_rewards` with **both** `community_id` (nullable) + `organization_id` (nullable)
   - FK: `community_id` → `organizations`
   - Index: `community_id`
   - ⚠️ `drop_community_id_from_tables` up() drops community_id unconditionally

3. `2026_05_15_000001_create_loops_table.php`
   - Creates `loops` with `community_id` (**NOT NULL**) — **no** `organization_id`
   - FK: `community_id` → `organizations`
   - Unique: `[community_id, slug]`
   - ⚠️ `drop_community_id_from_tables` up() drops community_id unconditionally

4. `2026_05_24_220000_add_organization_id_to_loops_table.php`
   - Adds `organization_id` (nullable) to loops
   - Copies `community_id` → `organization_id` via UPDATE
   - down(): SQLite guard, drops org_id on non-SQLite

5. `2026_05_28_000002_drop_community_id_from_tables.php`
   - Loops section: dropUnique `[community_id, slug]`, dropFK, dropColumn, add org FK, add `[org_id, slug]` unique
   - Referrals section: dropIndex, dropUnique, dropFK, dropColumn, add org FK, rename unique
   - Referral_rewards section: dropIndex, dropFK, dropColumn, add org FK
   - down(): re-adds community_id for all three (same blog_posts trade-off)

## Required changes if removing community_id

### Create migrations (3 files)

**`create_referrals_table.php`**:
- Remove `$table->uuid('community_id')->nullable()->index()`
- Remove `$table->foreign('community_id')->references('id')->on('organizations')->cascadeOnDelete()`
- Remove `$table->unique(['community_id', 'referrer_user_id', 'referred_user_id'], 'referrals_unique_pair')`
- **Add** `$table->unique(['organization_id', 'referrer_user_id', 'referred_user_id'], 'referrals_unique_pair')`

**`create_referral_rewards_table.php`**:
- Remove `$table->uuid('community_id')->nullable()->index()`
- Remove `$table->foreign('community_id')->references('id')->on('organizations')->cascadeOnDelete()`

**`create_loops_table.php`**:
- Remove `$table->uuid('community_id')` (NOT NULL)
- Remove `$table->foreign('community_id')->references('id')->on('organizations')->onDelete('cascade')`
- Remove `$table->unique(['community_id', 'slug'])`
- **Add** `$table->uuid('organization_id')->nullable()->index()` — org_id must be added to create
- **Add** `$table->foreign('organization_id')->references('id')->on('organizations')->onDelete('cascade')`
- **Add** `$table->unique(['organization_id', 'slug'])`

### Transition migration `add_organization_id_to_loops_table.php`
- No changes needed. up() has `if (Schema::hasColumn('loops', 'organization_id')) { return; }` guard — on fresh install after retrofit, it exits immediately. Correct.

### Transition migration `drop_community_id_from_tables.php`
- **Must add hasColumn guards** for loops, referrals, referral_rewards sections — same pattern as blog_posts (and the 4 simple tables)
- Without guards: up() will crash with "Column not found" on fresh install after retrofit
- down(): re-adds community_id unconditionally — same blog_posts compromise (noisy but safe)

## Rollback risks

Same profile as RUN-005G (blog_posts):

1. **`drop_community_id_from_tables` down()** re-adds community_id to loops/referrals/referral_rewards even on fresh install where it never existed. Accepted trade-off — migration cannot distinguish fresh-from-legacy after up().

2. **`add_organization_id_to_loops_table` down()** drops `organization_id` on non-SQLite. On fresh install after retrofit, this would drop a column that `create_loops` now owns. Mitigation: rolling back past `create_loops` drops the table entirely.

3. **No data loss risk** — all changes are schema-only (column addition/removal), no data migrations are affected.

## SQLite concerns

- `add_organization_id_to_loops_table` down() already returns early on SQLite (cannot drop columns).
- `drop_community_id_from_tables` down() uses direct column drops — SQLite PDO driver supports this via `ALTER TABLE ... DROP COLUMN` as of 3.35.0 (bundled with modern PHP). Verified: same pattern used for simple tables.
- Loops create migration uses `dropIfExists` for down() — no issue.

## PostgreSQL concerns

- None specific — all operations are standard column/constraint DDL.

## App code risk

- `loops`, `referrals`, `referral_rewards` may reference `community_id` in app business logic (queries, scopes, policies, Livewire). Fresh-install schema change removes the column entirely — app code referencing it would break on fresh installs.
- ⚠️ **Must verify app code references before merge.** This is a fresh-install-only concern — existing DBs keep community_id via the transition migrations.

## Summary

| Concern | Risk | Mitigation |
|---------|------|------------|
| drop_community_id lacks hasColumn | HIGH — will crash on fresh | Add hasColumn guards (proven pattern) |
| loops create lacks org_id | HIGH — must add to create | Straightforward column+FK+unique add |
| Rollback re-adds community_id | LOW — noisy but safe | Documented, same as RUN-005G |
| App code references community_id | MEDIUM — breaks fresh install | Audit before merge |
| SQLite | LOW | Guards already present |
| PostgreSQL | NONE | Standard DDL only |
